# Purpose
Transform unclear incoming requests into structured, decision-ready requirement intake outputs.
Use before discovery, estimation, commitment, sprint planning, or solutioning.

---

# When to Use
Use when input includes:
- Raw client messages or forwarded emails
- Workshop or meeting notes
- Unclear or urgent feature requests
- Partial scope ideas
- Stakeholder asks without success metrics or context
- Requests needing PM triage

# Do Not Use
Do not use for:
- Writing user stories
- Detailed solution architecture
- Sprint planning or final estimates
- Polished PRDs
- Technical implementation design

Route those to later-stage skills.

---

# Operating Principles
1. Clarify before committing.
2. Separate problem from requested solution.
3. Prefer concise business language.
4. Surface uncertainty explicitly — if unclear, say unclear.
5. Avoid assumptions unless necessary.
6. Flag delivery and commercial risks early.
7. Optimize for next action.

---

# Required Workflow
Read the request once, then produce each section below in order.

## 1. Request Summary
Rewrite the ask in one clear sentence.

## 2. Likely Business Goal
Infer the probable business outcome. Examples:
- Improve conversion
- Reduce manual work
- Improve reporting
- Increase retention
- Meet compliance
- Speed operations

## 3. User / Stakeholder Need
Who likely needs this and why.

## 4. What Is Clear
List confirmed facts from the input only. No inferences here.

## 5. Missing Information
List the highest-value unknowns only. Maximum 7 bullets.

## 6. Risks / Concerns
Include relevant risks such as:
- Unclear scope
- Unrealistic timeline
- Unknown dependencies
- Data unavailability
- Conflicting stakeholders
- Hidden change request
- Missing success metric

## 7. Intake Classification
Choose exactly one:
- Ready for Discovery
- Needs Clarification
- Likely Change Request
- Needs Technical Review
- Low Priority / Unclear Value
- Urgent Business Request

## 8. Recommended Next Step
Suggest the single best immediate PM action. Examples:
- Run 30-min discovery call
- Confirm users and success metrics
- Request current workflow documentation
- Involve engineering lead
- Assess timeline impact
- Defer to backlog review

# Style Rules
- Be concise. No long essays, no generic filler, no overconfidence.
- Use bullets only where they aid clarity.
- If something is unclear, say so explicitly.
- Prefer practical PM judgment over theory.
